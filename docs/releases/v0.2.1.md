# spec2openapi v0.2.1

[한국어 릴리즈 노트](v0.2.1.ko.md)

A robustness and correctness release for the **Swagger 2.0 → OpenAPI 3.x**
upgrader. Real-world Swagger documents are full of small omissions and
technically-invalid constructs; this release makes the upgrader turn those
into **valid OpenAPI 3** wherever a deterministic fix exists, and **fail with
a clear error** where conversion is genuinely impossible — instead of
silently producing a broken or lossy spec. Every automatic fix is recorded in
the output's `x-s2o` report.

## Highlights

**Valid OpenAPI 3, even from imperfect input.** The upgrader now enforces the
context-dependent rules that trip up naive conversions:

- `in: path` parameters are forced to `required: true`; path keys get a
  leading `/`; `{template}` segments with no parameter get one injected.
- `type: array` schemas without `items` get `items: {}`; a JSON-Schema
  `type` array is collapsed to a single type + `nullable` for 3.0.
- Invalid `securitySchemes` (unknown type, `apiKey` without `name`/`in`,
  `oauth2` flow missing its URL) are dropped rather than emitted; a partial
  `info` (missing `title`/`version`) is completed.
- `definitions` names with spaces or slashes are sanitized to valid
  component keys, with every `$ref` rewritten to match.
- `allowEmptyValue` is kept only on query params; `collectionFormat` maps to
  a location-legal `style`; a `tag` without a `name` is dropped.

**Fail loudly instead of silently.** A new exported `ConversionError`
(subclass of `ValueError`) is raised when a source cannot be faithfully
converted — an XSD element with an unresolvable type, or a WSDL with zero
convertible SOAP operations. The CLI prints a one-line `error: …` and exits 2
rather than dumping a traceback or returning an empty spec.

**Explicit and auditable output.** Every converted parameter now carries an
explicit boolean `required`. Every default, coercion, drop, or rename is
recorded in `x-s2o.assumptions` / `x-s2o.lossy`, so nothing changes silently.

## Behavior changes worth noting

- Output specs differ slightly from 0.2.0 (explicit `required`, normalized
  paths, sanitized component keys) — all valid OpenAPI 3, same semantics.
- Conversions that previously "succeeded" with an empty or subtly-invalid
  spec may now raise `ConversionError`. Catch it as `ValueError` if you
  script around the library.

## Install

```bash
pip install spec2openapi          # converter + CLI
pip install "spec2openapi[mcp]"   # + SOAP bridge (required to serve SOAP)
```

## What's Changed

- Force `in: path` parameters to `required: true` (#16)
- Drop invalid `securitySchemes` instead of emitting them (#18)
- Complete partial `info` and inject missing path parameters (#20)
- Respect parameter location (`allowEmptyValue`, `collectionFormat`) (#22)
- Harden the upgrader against non-conformant input (#24)
- Raise `ConversionError` instead of silently skipping unconvertible input (#26)
- Record previously-silent safe defaults and skips (#28)
- Always emit an explicit `required` on converted parameters (#30)
- Enforce leading-slash paths, named tags, and array `items` (#32)
- Sanitize Swagger definition names to valid component keys (#34)
